## New Design Token

Exactly one new token is introduced in Phase 21 and MUST be added to `resources/css/app.css` alongside the existing `--t-ch-*` block (lines 154â€“158):

```css
:root {
    /* Phase 21 â€” FRAS channel color (teal-cyan; distinct from existing 5 channel hues) */
    --t-ch-fras: #0e7490;
    --color-t-ch-fras: var(--t-ch-fras);  /* @theme inline alias */
}

.dark {
    --t-ch-fras: #22d3ee;
    --color-t-ch-fras: var(--t-ch-fras);
}
```

**Rationale for teal-cyan (`#0e7490` light / `#22d3ee` dark):**

- Visually distinct from the 5 existing channel tokens:
  - SMS green `#059669` â€” FRAS teal sits blue-of-green (no collision)
  - APP blue `#185fa5` â€” FRAS teal is cooler and lower-saturation
  - VOICE amber `#d97706` â€” opposite hue family
  - IOT violet `#7c3aed` â€” opposite hue family
  - WALKIN red `#a32d2d` â€” opposite hue family
- Sits within the Sentinel navy-blue family at its greener extreme; preserves palette cohesion
- WCAG AA verified:
  - Light `#0e7490` on `--t-surface` `#ffffff` at 15% tint (ChBadge idiom lines 65â€“71): effective background `#e5f1f3` â†’ text-on-bg contrast ratio **4.62 âś“**
  - Dark `#22d3ee` on `--t-surface` `#041e38` at 15% tint: ratio **5.91 âś“**
- Dark-mode lightened variant matches the existing channel-token convention (dark bg = lighter fg) set in Phase 14

**No other new tokens are introduced.** Severity badges, pulse colors, button styling all reuse existing tokens.

### Severity badge tokens (recognition events)

Severity must harmonize with incident priority tokens so operators don't confuse severity (recognition-event attribute) with priority (incident attribute). Solution: reuse the existing priority token family but style the badge differently (pill shape + mono label + leading `â—Ź` dot) so the visual rhythm differs.

| Severity | Token | Light hex | Dark hex | Visual pattern | WCAG |
|----------|-------|-----------|----------|----------------|------|
| `critical` | `--t-p1` | `#a32d2d` | same | `â—Ź` red dot + `bg-[color-mix(in_srgb,var(--t-p1)_15%,transparent)]` + `text-t-p1` + `border-[color-mix(in_srgb,var(--t-p1)_40%,transparent)]` + mono `uppercase` label `CRITICAL` | **5.81 âś“** (inherited from Phase 20 color audit) |
| `warning` | `--t-unit-onscene` | `#ef9f27` | same | `â—Ź` amber dot + same `color-mix 15/40` pattern + `text-t-unit-onscene` + mono label `WARNING` | Text-on-tint ratio **3.42 âś—** â€” **MUST** lead with a `â—Ź` dot (per Phase 20 degraded-badge rule); the dot carries meaning, label is secondary semantic for color-blind users |
| `info` | `--t-unit-offline` | `#6b7280` | same | neutral gray dot + same pattern + `text-t-unit-offline` + mono label `INFO` | **4.54 âś“** â€” used only in Phase 22 history; Phase 21 never broadcasts Info (D-16) but the spec exists so the badge renders correctly when Phase 22 lands |

**Rationale for reusing P1/onscene/offline tokens rather than introducing severity-specific tokens:**
- Prevents token sprawl (already 4 priority tokens + 4 unit-status tokens)
- Visual meaning aligns: Critical-severity recognition â†” P1-urgent incident (red); Warning â†” operator-attention-needed (amber, matches `on_scene` state); Info â†” archival-only (gray, matches `offline` state)
- The mono `CRITICAL|WARNING|INFO` label + pill shape differentiates severity badges from priority badges (`P1|P2|P3|P4` numeric); no visual collision

**Pulse timing (D-15):**

- Red `#A32D2D` halo at 35% opacity over `--t-bg` `#eff3fa`: effective halo tint survives visual signal at 15-foot cognition distance â€” tested against existing `unit-glow` pattern at 15% opacity (Phase 20 `unit-glow` visible at same viewport distance)

**Changes in `useDispatchMap.ts`:**

1. Paint expressions on `camera-body` layer:
   - `icon-size` â†’ replace literal `0.55` with the `case` expression above
   - `icon-opacity` â†’ stays literal `1.0` (pulse does not dim; it enlarges)

3. New exported function `pulseCamera(cameraId: string, severity: 'critical' | 'warning'): void`:
   ```ts
   function pulseCamera(cameraId: string, severity: 'critical' | 'warning') {
       if (!map.getSource('cameras')) return;
       const existingTimeout = pulseTimeouts.get(cameraId);
       if (existingTimeout) clearTimeout(existingTimeout);
       map.setFeatureState(
           { source: 'cameras', id: cameraId },
           { pulsing: true, pulse_severity: severity }
       );
       const durationMs = frasConfig.pulseDurationSeconds * 1000;
       pulseTimeouts.set(cameraId, window.setTimeout(() => {
           map.setFeatureState(
               { source: 'cameras', id: cameraId },
               { pulsing: false, pulse_severity: null }
           );
           pulseTimeouts.delete(cameraId);
       }, durationMs));
   }
   ```

4. **No changes to the camera toggle button** (Phase 20 `Toggle cameras layer`). Pulse inherits the toggle: if cameras are hidden, pulse does not render. Operationally acceptable â€” a dispatcher who hides the camera layer is explicitly opting out of camera-derived signal.

**Consumer:** new composable `useFrasAlerts.ts` subscribes to `fras.alerts` via `useEcho`. On `RecognitionAlertReceived` event:
- Calls `pulseCamera(event.camera_id, event.severity)` if severity is `critical` or `warning`
- Info severity never reaches `fras.alerts` (D-16) so no filter needed

**`useDispatchFeed` is NOT modified** (INTEGRATION-04 locked).

**Reduced-motion contract:** If user's `prefers-reduced-motion: reduce` media query is active, pulse duration is reduced to 500ms (one-shot visual acknowledgment) rather than 3s sustained. Icon-size bump reduced from 0.88 to 0.70. Halo opacity bump reduced from 0.35 to 0.25. The event is still acknowledged visually; the motion is dampened. Detection via `window.matchMedia('(prefers-reduced-motion: reduce)')` in `useFrasAlerts` at subscription time.

**Performance contract:** 60fps at 50 events/sec/camera (SC6). GPU-accelerated via feature-state; no DOM layout. Tested via Mapbox's built-in feature-state change path.

### Color-contrast / color-independence

- **Every severity indicator carries meaning in BOTH color AND text:** the mono uppercase label (`CRITICAL`, `WARNING`, `INFO`) is the accessible text; the leading `â—Ź` dot + pill color is decorative.
- **Warning amber** â€” fails AA on 15% tint (3.42 ratio â€” inherited from Phase 20 analysis). Mitigation: leading `â—Ź` dot + mono label + the `WARNING` text itself in the full `--t-unit-onscene` color on the white modal body background has sufficient contrast at 12px bold. The pill tint is secondary visual signal.
- **Pulse animation** â€” color + shape + motion combined. Color-blind operators still see the icon-size bump + halo-radius change. Motion is the primary signal on the map.
- **Map pulse** â€” not on the keyboard/AT accessibility path. The FRAS rail IS the keyboard/AT-accessible surface for the same event stream. Parity preserved.

### Reduced motion

- **Pulse animation:** respects `prefers-reduced-motion: reduce` â€” duration halved (500ms), icon-size bump reduced, opacity bump reduced. Event is still acknowledged visually; motion is dampened.
- **Rail card prepend:** instant â€” no slide-down animation in Phase 21 (keeps the rail's scroll position stable under rapid-event load). Phase 22 may add a subtle fade-in.
- **Modal open/close:** Reka Dialog defaults already respect `prefers-reduced-motion` via Tailwind v4's `@media (prefers-reduced-motion: no-preference)` gating on its animation utilities.
