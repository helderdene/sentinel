# Phase 17: Code Review Report

**Reviewed:** 2026-04-21
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

The gap-closure (plan 17-04) adds a single PDF download endpoint for resolved incident reports with a closure-based Gate, a role-middleware-gated route, a new controller action, a conditional header button on the incident detail page, and a 10-case Pest feature suite covering the 200/404/403/302 matrix.

Overall the implementation is clean, idiomatic, and well-tested. It follows IRMS conventions (closure gates only, no policies, explicit return types, curly-brace bodies, Wayfinder-imported action functions in the Vue layer) and the two-layer authorization pattern (role middleware as a coarse filter, Gate as the per-incident decision) is correctly composed. The raw DB pivot query in the Gate is intentionally scoped without `wherePivotNull('unassigned_at')` so ex-assigned responders retain post-resolution access — this is consistent with the spec and exercised by the "even with unassigned_at set" test.

Path-traversal risk on `Storage::disk('local')->download($incident->report_pdf_url, ...)` is low: `report_pdf_url` is written only by `GenerateIncidentReport::handle()` from a deterministic `incident-reports/{incident_no}.pdf` template where `incident_no` is generated by the `Incident::booted()` hook (format `INC-YYYY-NNNNN`). The column is mass-assignable, however, which is a latent risk worth noting rather than blocking (see WR-02).

Two warnings and five info items are listed below. No critical issues.

## Warnings

### WR-01: Download filename uses unsanitized `incident_no` in Content-Disposition header

**File:** `app/Http/Controllers/IncidentController.php:185-188`
**Issue:** The second argument to `Storage::download()` is `"{$incident->incident_no}.pdf"` and is emitted into the `Content-Disposition: attachment; filename=...` response header. `incident_no` is set by the model's `booted()` hook to the safe `INC-YYYY-NNNNN` format, so the current risk is zero — but the controller does not enforce that format at the boundary. If any future code path writes a non-standard `incident_no` (seeders, data import, admin UI), a newline or quote character could lead to header injection or a broken response header. Defense-in-depth: sanitize at the point of use.
**Fix:**
```php
return Storage::disk('local')->download(
    $incident->report_pdf_url,
    preg_replace('/[^A-Za-z0-9_\-]/', '_', $incident->incident_no).'.pdf',
);
```

### WR-02: `report_pdf_url` is mass-assignable; relies solely on controller/job discipline for path safety

**File:** `app/Models/Incident.php:55` (referenced from `app/Http/Controllers/IncidentController.php:185-188`)
**Issue:** `report_pdf_url` is listed in `$fillable` on the `Incident` model. Today only `GenerateIncidentReport::handle()` writes it, via `$incident->update(['report_pdf_url' => $path])` with a deterministic `"incident-reports/{$incident->incident_no}.pdf"` template — so path traversal is not currently reachable. However, because the attribute is mass-assignable, any future form request or controller that accidentally passes it through `$request->validated()` or `fill()` could allow an authenticated privileged user to set arbitrary paths (e.g., `../../../.env`), and `Storage::disk('local')->download()` would serve the file.

Because the file is served from `storage/app/private/` and `Storage::disk('local')->exists()` does not reject traversal components, this is a latent control gap. Not blocking for 17-04 (no such user-input path exists), but harden one of the two layers:

**Fix (controller-side, preferred — cheap defense-in-depth):**
```php
abort_if($incident->report_pdf_url === null, 404);
abort_unless(
    str_starts_with($incident->report_pdf_url, 'incident-reports/')
        && ! str_contains($incident->report_pdf_url, '..'),
    404,
);
abort_unless(Storage::disk('local')->exists($incident->report_pdf_url), 404);
```

**Fix (model-side, alternative):** Remove `report_pdf_url` from `$fillable` and give `GenerateIncidentReport` a dedicated setter (e.g., `$incident->report_pdf_url = $path; $incident->save();` with the attribute in `$guarded`-by-exclusion, or a small model method `markReportGenerated(string $incidentNo)` that builds the path internally).

## Info

### IN-01: Missing PHPDoc `@throws` annotation on `downloadReport`

**File:** `app/Http/Controllers/IncidentController.php:175-189`
**Issue:** `Gate::authorize()` throws `AuthorizationException` (→ 403) and `abort_if`/`abort_unless` throw `HttpException` (→ 404). The IRMS code style prefers PHPDoc over inline comments; other methods in this file document behavior but not thrown exceptions. Callers reading the signature only see `StreamedResponse`.
**Fix:**
```php
/**
 * Stream the generated incident report PDF for download.
 *
 * @throws \Illuminate\Auth\Access\AuthorizationException
 * @throws \Symfony\Component\HttpKernel\Exception\HttpException
 */
public function downloadReport(Incident $incident): StreamedResponse
```

### IN-02: Gate's `DB::table('incident_unit')` query duplicates relationship logic

**File:** `app/Providers/AppServiceProvider.php:179-182`
**Issue:** The responder branch uses a raw `DB::table('incident_unit')` existence check. This is fine, but `Incident::assignedUnits()` already exists as a `belongsToMany` relationship (confirmed at `app/Models/Incident.php:212`). Using the relationship keeps intent explicit and survives future pivot-table renames. The raw query intentionally omits `wherePivotNull('unassigned_at')` — add a comment to preserve that intent.
**Fix:**
```php
if ($user->role === UserRole::Responder) {
    if ($user->unit_id === null) {
        return false;
    }

    // Post-resolution access: include units whose assignment has ended
    // (do NOT filter wherePivotNull('unassigned_at')).
    return $incident->assignedUnits()
        ->wherePivot('unit_id', $user->unit_id)
        ->exists();
}
```
Either form is acceptable; if the raw query is kept for performance reasons, add a one-line comment explaining why `unassigned_at` is not filtered so a future reader does not "fix" it.

### IN-03: Test helper functions leak into the global namespace

**File:** `tests/Feature/Incidents/DownloadReportTest.php:17-43`
**Issue:** `makeResolvedIncidentWithReport()` and `makeResolvedIncidentWithMissingFile()` are declared as top-level `function ...` in a Pest test file, which registers them as globals the first time the file is autoloaded. If another Pest file in `tests/Feature/Incidents/` later defines a helper with either name, PHP will fatal. Pest's idiomatic fix is either `dataset()`, a closure assigned via `beforeEach`, or scoping helpers into `tests/Pest.php` / `tests/TestCase.php`.
**Fix:** Either (a) inline the two helpers into each test (they are only called 1-2 times), or (b) move them to `tests/Pest.php` with a clearly namespaced prefix like `downloadReportTest_makeResolvedIncidentWithReport()`, or (c) convert to a Pest dataset.

### IN-04: `makeResolvedIncidentWithMissingFile` issues an extra query for a value it already has

**File:** `tests/Feature/Incidents/DownloadReportTest.php:32-43`
**Issue:** `$incident = Incident::factory()->create([...])` then `$incident->update([...])` then `return $incident->fresh();`. The `->fresh()` call reloads the model from DB, which is only needed because `update()` was called with a single column and the caller may need the full reload. The same pattern in `makeResolvedIncidentWithReport()` is consistent. Minor — not a bug, just one extra query per test. Can be replaced with `$incident->refresh()` or removing `->fresh()` entirely since Eloquent already updates the in-memory attribute after `update()`.
**Fix:**
```php
$incident->update(['report_pdf_url' => 'incident-reports/missing.pdf']);

return $incident;
```

### IN-05: Missing test: resolved responder whose unit was never assigned to this incident (edge-of-pivot)

**File:** `tests/Feature/Incidents/DownloadReportTest.php`
**Issue:** The 10-case matrix covers: 4 privileged roles happy paths, assigned responder (with `unassigned_at` set), unrelated responder, responder with `unit_id = null`, null `report_pdf_url`, missing file, and unauthenticated redirect. A useful additional case — responder whose unit is attached to a *different* incident but whose `unit_id` matches — is not covered; the "unrelated responder" test uses a responder whose unit is attached to no incident at all. Consider adding:

```php
it('responder whose unit is assigned to a different incident receives 403', function () {
    $targetIncident = makeResolvedIncidentWithReport();
    $otherIncident = makeResolvedIncidentWithReport();
    $unit = Unit::factory()->create();
    $dispatcher = User::factory()->create(['role' => UserRole::Dispatcher]);
    $otherIncident->assignedUnits()->attach($unit->id, [
        'assigned_at' => now(),
        'assigned_by' => $dispatcher->id,
    ]);
    $user = User::factory()->create([
        'role' => UserRole::Responder,
        'unit_id' => $unit->id,
    ]);

    $this->actingAs($user)
        ->get(route('incidents.download-report', $targetIncident))
        ->assertForbidden();
});
```

This nails down the `where('incident_id', $incident->id)` clause in the Gate and would catch any accidental global-unit lookup regression.
