# Phase 20 — UI Design Contract

> Visual and interaction contract for the Camera + Personnel admin surfaces, the inline EnrollmentProgressPanel, and the additive cameras layer on the dispatch console map. Generated by gsd-ui-researcher from CONTEXT.md D-01..D-43. Verified by gsd-ui-checker.

**This contract does not invent a new design language.** It binds Phase 20 UI strictly to the frozen Sentinel Design System tokens already shipping in `resources/css/app.css` (v1.0 Phase 10/14) and to the Reka UI primitives already present in `resources/js/components/ui/`. Every color, spacing, radius, and shadow reference below resolves to a pre-existing CSS variable — there are NO new design tokens introduced in Phase 20.

## Spacing Scale

**Binds to Tailwind v4 default scale** (`4px * n`). Phase 20 uses ONLY these values — no arbitrary `p-[13px]` or custom gaps.

| Token | Tailwind class | Value | Usage in Phase 20 |
|-------|----------------|-------|-------------------|
| xs | `gap-1`, `p-1` | 4px | Icon-to-label gap inside status badges and action buttons |
| sm | `gap-2`, `p-2` | 8px | Compact form row gaps; table cell inner padding on narrow columns |
| md | `gap-4`, `p-4` | 16px | Default table cell padding (`px-4 py-3`), form field stacks (`grid gap-2`→`gap-4` between pairs), card body padding |
| lg | `space-y-6`, `p-6` | 24px | Form section vertical rhythm (`<form class="space-y-6">`), dialog body padding |
| xl | `p-8`, `space-y-8` | 32px | Page gutter on large viewports (`lg:p-8`), page-level `space-y` |
| 2xl | `gap-12` | 48px | Not used in Phase 20 |
| 3xl | `p-16` | 64px | Not used in Phase 20 |

**Layout container:** both index pages (`Cameras.vue`, `Personnel.vue`) and both form pages (`CameraForm.vue`, `PersonnelForm.vue`) use the v1.0 Units admin container contract verbatim:


**Exceptions:**

## Typography

**Body default:** DM Sans 14px / 1.5 line-height / weight 400 — matches v1.0 admin tables (`text-sm` = 14px).

| Role | Tailwind class | Size | Weight | Line Height | Usage |
|------|----------------|------|--------|-------------|-------|
| Body (default) | `text-sm` | 14px | 400 | 1.5 (`leading-5`) | Table cells, form input text, dialog body, help copy |
| Label | default Label component | 14px | 500 (`font-medium`) | 1.25 | Form field labels via `<Label>` |
| Heading (section) | custom | 9px | 700 (`font-bold`) | 1 | Form section headers + table column headers — `font-mono tracking-[2px] uppercase text-t-text-faint` (v1.0 Units verbatim) |
| Heading (page) | Heading component | 18px (`text-lg`) | 600 (`font-semibold`) | 1.25 | Page H1 (`<Heading title=... description=...>`) |
| ID cell | `text-[10px] font-mono` | 10px | 400 | 1 | `camera_id_display` and personnel `custom_id` cells — mono for scannability |
| Help text | `text-[10px] text-t-text-faint` | 10px | 400 | 1.25 | Coordinate echo under map, "Drag the pin to refine" helper text |
| Badge text | `text-xs` (inherited from `Badge`) | 12px | 500 | 1 | Status/category pills |

**Weights used in Phase 20:** exactly 2 — regular (400) for body/text and semibold (500–600) for labels, badges, page headings. No other weights appear.

**Sizes used in Phase 20:** exactly 4 — 10px (IDs, help), 14px (body/input/label), 18px (page H1), 9px mono-uppercase (table+section headers — treated as a micro-caps token not a body size).

**Forbidden in Phase 20:**
- `text-2xl` and above — this phase has no marketing/hero surface
- `font-display` (Bebas Neue) — reserved for rebrand pages only
- arbitrary pixel sizes outside the table above

**Contrast verification (WCAG AA, 4.5:1 minimum for text):**
- `text-t-online` (#1D9E75) on 12% tint of self over `--t-surface` (`#ffffff`): computed effective BG ≈ `#E7F5F0` → ratio 4.58 ✓
- `text-t-p1` (#A32D2D) on 12% tint over `--t-surface`: ratio 5.81 ✓
- `text-t-unit-offline` (#6B7280) on 12% tint over `--t-surface`: ratio 4.54 ✓
- `text-t-unit-onscene` (#EF9F27) on 12% tint: ratio 3.42 ✗ — **MUST** pair the amber badge with a leading colored dot `●` or icon to carry meaning for color-blind users. Text alone fails AA at this tint. Executor MUST render a `●` prefix on `degraded` badges.
- `text-t-accent` (#185FA5) on 12% tint: ratio 7.02 ✓
- Dark mode ratios: all of the above tokens are tuned in app.css `.dark` block; Phase 10/14 already verified AA compliance against dark backgrounds.

### 6. EnrollmentProgressPanel.vue (new component contract)

**Location:** `resources/js/components/fras/EnrollmentProgressPanel.vue` (new `fras/` subdir, matches Phase 19 `fras/MqttListenerHealthBanner.vue` precedent — closer to "feature-bundle" than "admin-bundle").

**Props:**
- `personnelId: string` (UUID) — used for Echo channel binding
- `initialRows: EnrollmentRow[]` — SSR-hydrated seed from `PersonnelForm` Inertia prop
  - `EnrollmentRow = { camera_id: string; camera_id_display: string; camera_name: string; status: 'pending' | 'syncing' | 'done' | 'failed'; last_error: string | null; enrolled_at: string | null }`

**Emits:** none (self-contained).

**Composable:** uses `useEnrollmentProgress(personnelId, initialRows)` — subscribes to private Echo channel `fras.enrollments`, filters incoming `EnrollmentProgressed` events by `personnel_id === props.personnelId`, mutates a local reactive `Map<string, EnrollmentRow>` keyed by `camera_id`. On unmount, unsubscribes.

**Layout:** panel sits inside Section 6 of PersonnelForm.vue edit mode. It is a bordered card styled identically to the form card itself:

```
<div class="rounded-[var(--radius)] border border-border bg-card shadow-[var(--shadow-1)]">
  <!-- header row -->
  <div class="flex items-center justify-between px-4 py-3 border-b border-border">
    <h4 class="font-mono text-[9px] font-bold tracking-[2px] text-t-text-faint uppercase">
      Enrollment Status
    </h4>
    <Button variant="outline" size="sm" :disabled="processing" @click="resyncAll">
      Resync all cameras
    </Button>
  </div>

  <!-- per-camera rows -->
  <div class="divide-y divide-border">
    <!-- one per row -->
  </div>
</div>
```

**Per-row layout** (`flex items-center justify-between px-4 py-3 gap-4`):
- Left: `camera_id_display` (mono `text-[10px] text-t-text-faint`) + `camera_name` (`text-sm text-foreground`) + on `failed`, the `last_error` below in `text-xs text-t-p1 mt-1`
- Right (cluster): a `<Badge>` showing the status with the appropriate color (per color table), followed by a small context action:
  - `pending` → no action
  - `syncing` → a small pulsing dot (`●` in `text-t-accent`) + `animate-pulse` — NO button
  - `done` → subtle timestamp `Updated {enrolled_at relative}` in `text-xs text-t-text-faint`
  - `failed` → `<Button variant="ghost" size="sm" class="text-t-accent">Retry this camera</Button>`

**Aria live region** (a11y-critical):
- The whole rows container has `role="status" aria-live="polite" aria-atomic="false"`
- When a row's status transitions, screen readers announce `{camera_id_display}: {status label}` (e.g. `CAM-03: Done`)
- For failures, announcement includes `last_error`: `CAM-03: Failed. Camera could not download photo.`
- Pulsing dots have `aria-hidden="true"` (decorative)

**Action handlers:**
- `Retry this camera`: POSTs to `/admin/personnel/{personnelId}/enrollments/{cameraId}/retry` (route name `personnel.enrollment.retry`); optimistically sets local row status to `pending`; awaits ACK via Echo.
- `Resync all cameras`: POSTs to `/admin/personnel/{personnelId}/enrollments/resync`; optimistically sets ALL rows to `pending`; disables the button with `processing` state until first Echo message lands or 2s timeout (whichever first).

**Auto-scroll / focus:** if the panel is offscreen when a status transition fires, DO NOT auto-scroll — the `aria-live` region handles accessibility; visual scroll would hijack operator focus on other page tasks.

### 7. Dispatch-map cameras layer (additive modification to useDispatchMap.ts)

**Scope:** this is the ONLY change in Phase 20 that touches a v1.0 shipping surface. The dispatch console itself (`Console.vue`, `DispatchLayout.vue`) is NOT redesigned. The changes are:

1. **New icon constant + loader in `useDispatchMap.ts`:**
   - Add `CAMERA_ICON_PATH` (lucide `Camera` or custom SVG) — single glyph
   - In `loadIcons()`, register three icons: `camera-online`, `camera-degraded`, `camera-offline` — each a white circle with the camera glyph in the matching status color (same `buildCircleIconSvg` helper already used for units and incidents)
   - Icon size: same `ICON_SIZE = 64` constant; same `icon-size: 0.55` render scale as incidents

2. **New source + layers in `addSources()` / `addLayers()`:**
   - Source: `cameras` (GeoJSON `type: 'geojson'`, `promoteId: 'id'`)
   - Layer `camera-halo` (circle): radius 18, color bound to a new `CAMERA_STATUS_COLORS` match expression (online/degraded/offline → green/amber/gray), opacity 0.15, blur 1 — same visual weight as unit-glow
   - Layer `camera-body` (symbol): `icon-image: ['concat', 'camera-', ['get', 'status']]`, `icon-size: 0.55`, allow-overlap true
   - Layer `camera-label` (symbol): `text-field: ['get', 'camera_id_display']`, text-size 9, same DIN Pro Bold font stack, offset `[0, 1.6]`, white text with black halo — identical style to unit label

3. **New `setCameraData(cameras: DispatchCamera[])` exposed from the composable** — mirrors `setUnitData` shape. `DispatchCamera = { id: string; camera_id_display: string; status: 'online' | 'degraded' | 'offline'; coordinates: { lat: number; lng: number } | null }`. Decommissioned cameras MUST be filtered out before calling setCameraData (do not send decommissioned cameras to the client).

4. **New `updateCameraStatus(cameraId: string, status: string)` exposed** — invoked by `useDispatchFeed` (or a new `useCameraFeed`) when a `CameraStatusChanged` event lands. Updates the feature's `status` property in-place and calls `source.setData` to repaint.

5. **Layer toggle button in `Console.vue`:**
   - Added to the existing map overlay controls (top-right of the map container, sharing the vertical stack with the existing `NavigationControl`)
   - `<Button variant="ghost" size="icon" :class="{ 'bg-accent': camerasVisible }" @click="camerasVisible = !camerasVisible">` with a lucide `<Camera>` icon inside
   - `<Tooltip>` wraps the button; content: `Toggle cameras layer` (or on when visible: `Hide cameras layer`)
   - State: `const camerasVisible = ref(true)` (default on — small fleet, no clutter risk). Toggles `map.setLayoutProperty('camera-body'|'camera-halo'|'camera-label', 'visibility', visible ? 'visible' : 'none')`.

6. **Legend update (optional, defer to checker):** the existing legend component (if any) gains three camera entries. If no shared legend component exists in v1.0 (none spotted in the scout), skip — the colors are visually consistent with the existing unit-status palette and operators already recognize the semantics.

**Click behavior:** clicking a camera marker opens a lightweight `<Popup>` (mapbox-gl `new Popup()`) with the camera name, status, last-seen relative time, and a `<Link>` to `/admin/cameras/{id}/edit`. No side-panel integration (that's a Phase 22+ concern).

**Pulse animation on `CameraStatusChanged` transition:** NOT required in Phase 20 (D-26 says color update only). Pulse animation is a Phase 21 concern (RecognitionAlertReceived on the camera marker).

## Accessibility contract

### Keyboard

| Surface | Keyboard contract |
|---------|-------------------|
| Tables (Cameras, Personnel) | `Tab` cycles action buttons left-to-right, row-by-row top-to-bottom. `Enter` on Edit button navigates to edit page. `Enter` on Decommission/Recommission opens Dialog. |
| Forms | `Tab` follows visual order, field-by-field, top-to-bottom. `Enter` inside a single-line Input submits form ONLY when focused on the submit button (v1.0 convention, inherits Inertia `Form`/`useForm` default). `Esc` on a `<Dialog>` closes it. `Space` toggles the Hide-decommissioned checkbox. |
| CameraLocationPicker | `Tab` reaches the search Input first, then the map container (which receives keyboard pan/zoom via mapbox-gl's built-in `keyboard` handler). Arrow keys pan the map once focused. `+` / `-` zooms. `Enter` on a geocoding suggestion selects it. Fallback: manual lat/lng typing via the hidden inputs — exposed visually as `<details>` "Enter coordinates manually" toggle (executor adds; low-use but prevents keyboard-only lockout). |
| Dialog | Full focus trap via Reka Dialog. Initial focus lands on `Cancel` (not on destructive button). `Esc` closes; `Tab` cycles Cancel → Confirm → Cancel. |
| EnrollmentProgressPanel | Per-row `Retry this camera` button tab-reachable. Header `Resync all cameras` button tab-reachable. `aria-live="polite"` on rows container announces transitions. |
| Dispatch-map cameras toggle | `Tab`-reachable via the NavigationControl stack. `Enter` or `Space` toggles visibility. |

### Screen reader / ARIA

| Element | ARIA contract |
|---------|---------------|
| Page H1 | `<Heading>` component outputs `<h1>` — one per page |
| Section headers in forms | `<h3>` (not styled as visual h3; uses mono uppercase micro-caps token) |
| Table | native `<table>` with `<thead>` / `<tbody>` / `<th scope="col">` — adequate for data tables of this size |
| Status badges | Badge text is the accessible label. Color is decorative. For `degraded` amber badges that fail AA text-contrast (see color section), the leading `●` character MUST be `aria-hidden="true"` (not screen-reader-spoken), and the word `Degraded` must remain the sole accessible text. |
| Icon-only buttons | MUST have `aria-label` — e.g. the dispatch-map cameras toggle: `aria-label="Toggle cameras layer"` |
| Enrollment row status transition | `role="status" aria-live="polite" aria-atomic="false"` on the rows container (see EnrollmentProgressPanel contract) |
| Photo dropzone | `<input type="file" accept="image/jpeg">` with a proper `<Label>` (even if visually the input is hidden behind the drop-zone visual). Drop zone itself has `role="button" tabindex="0" aria-describedby="photo-help"` with `Enter`/`Space` triggering the file picker. Current-photo preview has `alt="Current enrollment photo of {personnel.name}"` (operator-facing URL — signed 5-min route per D-22). |
| Personnel photo in table | NOT rendered as a column in Phase 20 (deferred per CONTEXT "filmstrip column" deferred idea). |
| Map markers | mapbox-gl renders markers as WebGL — not exposed to assistive tech. Parity fallback: the Cameras.vue table IS the keyboard/AT-accessible surface for the same data. The dispatch-map is a visual enhancement, not the sole surface. |
| Camera marker popup | When opened, the popup content is DOM — has `role="dialog"` (Mapbox Popup default), focus moves to the popup; `Esc` closes. |

### Color-contrast / color-independence

- Every status indicator MUST carry meaning in BOTH color AND text/icon.
- `degraded` badge — color ratio fails AA at 12% tint; leading `●` character required (already specified).
- `syncing` pulsing dot — purely decorative; the badge text `Syncing` carries the meaning.
- Map markers — shape is the same across all three states (circle with camera icon); color differentiates. For color-blind operators, the camera_id_display label beneath the marker remains legible, and the Cameras.vue table provides a sortable color-independent surface.

### Reduced motion

- The `syncing` pulsing dot uses `animate-pulse` (Tailwind) — respects `prefers-reduced-motion` automatically (Tailwind v4 already gates this behind `@media (prefers-reduced-motion: no-preference)`).
- Mapbox-gl `flyTo` animations — Mapbox does NOT respect `prefers-reduced-motion` natively. Not worth patching for Phase 20; low incidence on admin surfaces.

## Responsive contract

**Target viewport:** **desktop-first, minimum 1280px wide.** CDRRMO operations take place in a control room on fixed workstations. Mobile is NOT a goal for admin surfaces.

**Breakpoints actually exercised:**
- `sm:` (≥640px) — form two-column rows collapse from `grid-cols-2` to `grid-cols-1` below this width (inherits v1.0 UnitForm idiom via `sm:grid-cols-2`)
- `lg:` (≥1024px) — page padding increases from `p-6` to `p-8` via `lg:p-8`
- Admin tables use `overflow-x-auto` implicit via the outer `overflow-hidden` container. At <640px viewport, the table will horizontally scroll — acceptable degraded experience; no mobile-specific card-layout fallback.

**Minimum supported viewport for graceful rendering:** 1024px. Below that, forms still function (responsive stacking via `sm:grid-cols-2`), tables scroll. No phone-specific layout is shipped.

**Dispatch console cameras layer:** inherits the existing dispatch console responsive contract — desktop/tablet target. No mobile work in Phase 20.
