```markdown
# Laravel 13 Upgrade — Deploy Runbook

> **Scope:** IRMS Phase 17 (Laravel 12 → 13) production cutover.
> **Audience:** CDRRMO ops admin following this runbook without AI assistance.
> **Acceptance:** FRAMEWORK-03 — no queued job executes under a mixed Laravel-version worker.
> **Source:** `.planning/phases/17-laravel-12-13-upgrade/17-RESEARCH.md` §Horizon drain-and-deploy.

## 1. Purpose

This runbook is the reproducible procedure for deploying the Laravel 12 → 13 upgrade to
production. It exists because a mixed-version worker pool (L12 + L13 workers processing
the same Redis queue) can corrupt queued job payloads. The protocol drains Horizon
BEFORE deploying, so every job that executes post-deploy runs under L13 only.

## 2. Preconditions

- [ ] `main` branch passes full Pest suite green on L13 (CI badge or local `php artisan test --compact`)
- [ ] The 6 broadcast payload snapshots match byte-for-byte (CI badge or local `php artisan test --compact --filter=Broadcasting`)
- [ ] Production Supervisor has `irms-horizon` and `irms-reverb` programs configured with `stopwaitsecs=3600`
- [ ] Operator / admin has SSH access to the production DO droplet
- [ ] Active incident count < 10 (verify via ops dashboard; high-volume drains extend the cutover window)

## 3. Pre-Deploy: Drain Horizon

The drain must be `pause → poll-status → terminate`, not `terminate` alone. Pausing
stops Horizon from pulling new jobs into workers; polling waits for in-flight to
complete; terminate exits workers gracefully. (See RESEARCH.md Pitfall 5.)

```bash
# 1. Pause Horizon — stops pulling new jobs into workers (queued jobs stay in Redis)
php artisan horizon:pause

# 2. Wait for in-flight jobs to finish (status count goes to 0)
#    Poll with timeout (typical drain: < 30s for IRMS; P1 alerts may hold ~5s each)
for i in {1..60}; do
    php artisan horizon:status 2>&1 | grep -q 'inactive\|paused' && break
    sleep 1
done

# 3. Terminate — workers exit gracefully after current tick
php artisan horizon:terminate
```

## 4. Deploy

```bash
# 4. git pull / composer install / wayfinder:generate / optimize:clear
git pull
composer install --no-dev --optimize-autoloader
php artisan wayfinder:generate
php artisan optimize:clear
```

## 5. Post-Deploy: Restart + Verify

```bash
# 5. Supervisor restarts the horizon program automatically after horizon:terminate
#    (stopwaitsecs=3600 in supervisor conf gives up to 1h for drain)
sudo supervisorctl restart irms-horizon
sudo supervisorctl restart irms-reverb

# 6. Verify
php artisan horizon:status   # Should report 'running'
curl -sf https://irms.example.com/up   # Health check — replace domain as appropriate
```

## 6. Smoke Test (Phase 17 Success Criterion 4)

Spot-check the v1.0 dispatch flow end-to-end on the upgraded build:

1. Operator posts a P2 incident at `/intake`.
2. Dispatcher triages + assigns a unit at `/dispatch`.
3. Responder (second browser session) receives the assignment push, acknowledges within 90s.
4. Responder advances through Dispatched → Acknowledged → OnScene → Resolved with an outcome.
5. PDF incident report generates correctly.
6. Check `app/Events/*` broadcasts landed on dispatch + responder clients by watching
   the network tab for Reverb WebSocket frames — payload fields match Phase 17 Wave 1
   golden fixtures.

Expected time to green: ~5 minutes.

## 7. Rollback

Trigger rollback if ANY of the following is observed:

- Any failing Pest test post-deploy (monitoring / CI)
- Broadcast payload mismatch detected (dispatcher browser shows malformed event data)
- Queued job execution failure in Horizon dashboard

# === ROLLBACK ===
