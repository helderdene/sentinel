<task type="auto" tdd="true">
  <name>Task 1: IoT sensor webhook -- controller, HMAC middleware, config, and tests</name>
  <files>
    app/Http/Controllers/IoTWebhookController.php,

  <behavior>
    - POST /webhooks/iot-sensor with valid HMAC signature creates a PENDING incident with channel=iot
    - Sensor type maps to incident type code and priority from config('services.iot.sensor_mappings')
    - Payload includes: sensor_type, sensor_id, value, threshold, location_text, latitude, longitude, timestamp
    - Missing or invalid HMAC signature returns 401
    - Stale timestamp (>5 minutes old) returns 401
    - Unknown sensor_type returns 422
    - Incident's notes field includes sensor details: "IoT Alert: {sensor_type} sensor {sensor_id} reported value {value} exceeding threshold {threshold}"
    - If latitude/longitude provided, auto-assigns barangay via BarangayLookupService
    - Created incident has raw_message = JSON-encoded original payload
  </behavior>
  <action>
    1. Create `app/Http/Middleware/VerifyIoTSignature.php`. In the `handle()` method:
       - Read `X-Signature-256` header and `X-Timestamp` header from request
       - Reject with 401 if either header is missing or if `config('services.iot.webhook_secret')` is empty
       - Reject with 401 if `abs(time() - (int)$timestamp) > 300` (stale request)
       - Compute expected signature: `'sha256=' . hash_hmac('sha256', $timestamp . '.' . $request->getContent(), $secret)`
       - Reject with 401 if `!hash_equals($expected, $signature)`
       - Pass to next middleware
       Reference the code example in 02-RESEARCH.md.

    2. Register the middleware alias in `bootstrap/app.php`: add `'verify-iot-signature' => VerifyIoTSignature::class` to the alias array alongside the existing 'role' alias.

    3. Create `app/Http/Controllers/IoTWebhookController.php` as an invokable controller (`__invoke` method). Constructor injects BarangayLookupService. In `__invoke(Request $request)`:
       - Validate payload: sensor_type (required, string), sensor_id (required, string), value (required, numeric), threshold (required, numeric), location_text (nullable, string), latitude (nullable, numeric), longitude (nullable, numeric)
       - Look up sensor mapping from `config('services.iot.sensor_mappings')`. If sensor_type not found, return 422 JSON error.
       - Find IncidentType by code from mapping. If not found, return 422.
       - Create Incident with: incident_type_id, priority from mapping, status=PENDING, channel=IncidentChannel::IoT, location_text, notes="IoT Alert: {sensor_type} sensor {sensor_id} reported value {value} exceeding threshold {threshold}", raw_message=json_encode($request->all()).
       - If lat/lng provided, create Point and auto-assign barangay.
       - Create IncidentTimeline entry: event_type='incident_created', event_data=['source' => 'iot_sensor', 'sensor_type' => ..., 'sensor_id' => ...].
       - Return 201 JSON with incident_no and incident id.

    4. Add webhook routes to `routes/web.php`. Create a `Route::prefix('webhooks')` group at the TOP of the file (before the auth middleware group). Both webhook routes must exclude CSRF verification:
       ```php
       Route::prefix('webhooks')->group(function () {
           Route::post('iot-sensor', IoTWebhookController::class)
               ->middleware('verify-iot-signature')
               ->withoutMiddleware([\Illuminate\Foundation\Http\Middleware\VerifyCsrfToken::class])
               ->name('webhooks.iot-sensor');
       });
       ```

    5. Add `IOT_WEBHOOK_SECRET=test-iot-secret-key` to `.env.testing` for test environment.

    6. Create `tests/Feature/Intake/IoTWebhookTest.php` with Pest tests:
       - Test valid flood_gauge payload creates PENDING incident with correct type and P2 priority
       - Test valid fire_alarm payload creates P1 incident
       - Test missing HMAC signature returns 401
       - Test invalid HMAC signature returns 401
       - Test stale timestamp (>5 minutes) returns 401
       - Test unknown sensor_type returns 422
       - Test incident notes contain sensor details
       - Test raw_message preserves original JSON payload
       - Test auto-assigns barangay when coordinates provided (use known Butuan coordinate from Phase 1 spatial test)
       - Helper method in test to generate valid HMAC signature for payload

    Run `vendor/bin/pint --dirty --format agent` after PHP files are created.
  </action>
  <verify>

  </verify>

    - Filipino keywords (sunog, baha, aksidente, ambulansya, lindol) correctly map to incident types
    - English keywords (fire, flood, earthquake) also map correctly
    - Unmatched keywords default to General Emergency (PUB-001)
    - Raw SMS message preserved in raw_message field
    - Notes include: "SMS from {sender}: {message}"
    - Location text extracted from message (best effort) for geocoding
    - Auto-reply SMS is logged via StubSemaphoreSmsService (not actually sent)
    - Channel monitor: incident queue endpoint already returns channelCounts; this test verifies counts per channel
  </behavior>
  <action>
    1. Create `app/Services/SmsParserService.php`. Constructor takes no dependencies (reads config). Methods:
       - `classify(string $message): array` -- Returns `['incident_type_code' => string, 'matched_keyword' => string|null]`. Logic: lowercase the message, scan for any keyword in `config('sms.keyword_map')`. First match wins. If no match, return default_type_code from config and matched_keyword=null.
       - `extractLocation(string $message): ?string` -- Best-effort location extraction. Look for common patterns: "sa [location]", "at [location]", "near [location]", "dito sa [location]". Extract the text after the preposition up to the next comma, period, or end of message. Return null if no pattern found.
       - `parsePayload(array $payload): array` -- Extract sender (from 'sender' or 'from' key), message (from 'message' or 'body' key), timestamp. Normalize.

    2. Create `app/Http/Controllers/SmsWebhookController.php` as an invokable controller. Constructor injects SmsParserService, BarangayLookupService, SmsServiceInterface. In `__invoke(Request $request)`:
       - Parse payload via SmsParserService::parsePayload()
       - Classify message via SmsParserService::classify()
       - Extract location via SmsParserService::extractLocation()
       - Find IncidentType by code from classification
       - Create Incident with: incident_type_id, priority=incidentType.default_priority, status=PENDING, channel=IncidentChannel::Sms, caller_contact=sender, location_text=extracted location or null, notes="SMS from {sender}: {message}", raw_message=message
       - Create IncidentTimeline entry: event_type='incident_created', event_data=['source' => 'sms', 'sender' => ..., 'matched_keyword' => ...]
       - Send auto-reply via SmsServiceInterface::send() with message: "Your emergency report has been received. Incident #{incident_no}. Help is on the way."
       - Return 200 JSON with incident_no.

    3. Add SMS webhook route in the existing `Route::prefix('webhooks')` group in `routes/web.php`:
       ```php
       Route::post('sms-inbound', SmsWebhookController::class)
           ->withoutMiddleware([\Illuminate\Foundation\Http\Middleware\VerifyCsrfToken::class])
           ->name('webhooks.sms-inbound');
       ```
