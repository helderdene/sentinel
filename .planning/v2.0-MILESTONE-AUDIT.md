### Flow 1: Recognition ingestion (Phase 18 → 19 → 21 → 22)

End-to-end: MQTT RecPush → RecognitionHandler persists to recognition_events (18 schema) → RecognitionToIntakeBridge classifies severity → /fras/alerts feed broadcasts to operator.

Evidence: Live cloud-broker UAT (19-UAT tests 7-10) verified full chain against real hardware (cloud broker 148.230.99.73, deviceId=irms-smoketest-01). Row persisted, images written to fras_events disk, severity classified, alert visible.

### Flow 2: Camera heartbeat → dispatch console health (19 → 20)

HeartbeatHandler bumps cameras.last_seen_at → CameraStatusWatchdog flips to online/degraded/offline → broadcasts CameraStatusChanged → dispatch console camera panel + admin /admin/cameras update live.

Evidence: 19-VERIFICATION truth 1 (TopicRouter + HeartbeatHandler), 20-VERIFICATION (CameraStatusChanged broadcast), G-03 live-update gap resolved.

### Flow 3: Personnel enrollment → camera sync (20 → 19)

Admin uploads personnel photo → FrasPhotoProcessor validates/hashes → EnrollPersonnelBatch enqueued per active camera (WithoutOverlapping) → camera HTTP-fetches photo via unguessable-UUID URL → AckHandler correlates ACK → URL revoked.

Evidence: 20-VERIFICATION PERSONNEL-02 through PERSONNEL-07 all verified with unit + feature tests.

### Flow 4: Recognition event → Incident promote → responder context (22 → v1.0 responder)

Operator at /fras/events opens detail modal → PromoteIncidentModal submits reason+priority → FrasEventHistoryController::promote creates Incident → responder SceneTab shows POI accordion (face + personnel + camera + timestamp), never raw scene image.

Evidence: 22-VERIFICATION truth 3 (POI accordion, scene image absence enforced by arch test), ResponderSceneTabTest 6/6 passed.
